# Sanctuary — Phase 1 Greybox Slice

AI-assisted, solo-dev build in **Godot 4.x**. Phase 0 (the harness) and Phase 1
(the greybox vertical slice) are done: the project boots into one grey room that
exercises every core system crudely — first-person movement, look-to-interact,
item pickup, a locked door gated on an item, a branching conversation, and a
patrolling guard with line-of-sight detection. Claude Code has a feedback loop it
can read end to end (`make test`, `make smoke`).

No final art yet — everything is greybox primitives built in code. Phase 2 starts
authoring real "Ceremony" content on top of these systems.

## Prerequisites

- **Godot 4.x** — uses the Compatibility (OpenGL) renderer, a good fit for the
  PSX look and low-spec targets.
- *(optional)* **gdtoolkit** for linting/formatting: `pip install gdtoolkit`
- *(optional, later)* **GUT** for richer tests — see `CLAUDE.md`.

## First run

1. Import once so the `.godot/` cache and global class registry are built —
   either open the project in Godot (`File > Open Project` → `project.godot`) or
   run `godot --headless --path . --editor --quit`. **This step is required**: on
   a fresh checkout `.godot/` is absent, so `class_name` scripts aren't
   registered and headless `make test`/`make smoke` will fail until it's done. If
   prompted about an engine-version mismatch, letting it open/convert is fine.
2. From a terminal in the project root:
   - `make smoke` — boots the core loop **and drives the greybox slice** headless
     (look at the locked door, pick up the key, open it, branch a conversation,
     get spotted), asserts, exits 0 on success.
   - `make test` — runs the unit tests headless.
   - `make run` — launches the greybox slice (human only; mouse-look needs a
     window). **Controls:** WASD move, mouse look, **E** interact / talk / open,
     **Ctrl** crouch, **Esc** quit.

   If `godot` isn't on your PATH, pass the binary:
   `make smoke GODOT=/path/to/Godot_v4.x`
3. *(optional)* Normalise code style once so `make check` passes cleanly:
   `gdformat scripts tests`. After that, `make check` (lint + format + test +
   smoke) is your pre-commit gate.

## Layout

Full contract is in `CLAUDE.md`. Short version:

- `scripts/core/` — autoload singletons: `GameState`, `SaveSystem`,
  `SceneManager`, `AudioDirector`.
- `scripts/inventory/` — inventory autoload + JSON item catalog.
- `scripts/player/` — first-person controller + camera interaction ray.
- `scripts/interaction/` — `door`, `pickup`, `npc_talker` (the interactable group).
- `scripts/dialogue/` — `DialogueRunner` (pure JSON walker) + dialogue UI.
- `scripts/stealth/` — `GuardVision` (pure cone maths) + the patrolling `guard`.
- `scripts/levels/` — `room_builder` (assembles a room from data).
- `scripts/ui/` — interaction prompt HUD.
- `data/` — the content layer: `items/`, `dialogue/` (`*.json` conversations),
  `rooms/` (`greybox.json`).
- `scenes/entities/`, `scenes/ui/`, `scenes/levels/` — generated by
  `tools/godot/scaffold_phase1.gd`; `scenes/core/` boots into the greybox.
- `tests/` — dependency-free headless test runner + the slice-driving smoke scene.
- `assets/` — **binary, human/Blender-owned**; AI does not touch this.

## Next: Phase 2 (Stage 1 "Ceremony" content)

Phase 1's systems are crude but complete. Phase 2 authors real content as **data**
on top of them — items in `data/items/`, conversations in `data/dialogue/`, rooms
in `data/rooms/` — and (per `CLAUDE.md`) swaps the temporary in-house
`DialogueRunner` for the Dialogue Manager addon. The greybox room and its grey
primitives stay as a reference harness; new rooms are new JSON, not new code.

What still wants a human's eyes (the agent can't see the running game): tune the
player feel, room layout, and dialogue-UI look in-editor via `make run` and the
`@export` sliders on the generated scenes.
